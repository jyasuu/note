# 全覽地圖（先看這個）

| 類型               | Anti‑pattern        |
| ---------------- | ------------------- |
| 檔案層              | 小檔案爆炸               |
| Partition        | 過度 / 錯誤 partition   |
| Commit           | 每筆 / 高頻 commit      |
| Streaming        | micro‑batch misuse  |
| Concurrent write | 多 writer 沒協調        |
| Schema           | 隨意 schema evolution |

| Operational      | 沒做 maintenance      |

## 1️⃣ **Small Files Anti‑Pattern（最常見、最致命）**

### 壞現象

*   每次寫入產生大量 <10MB 的 Parquet
*   Query 慢、planning time 長、metadata 爆炸

### 為什麼？

Iceberg **允許 small files，但不會幫你消滅它**

寫入模型是：

> 「append data files → commit snapshot」

❌ 它不會自動 merge

### 壞例子

```text
每分鐘一個 Spark job
→ 每次寫 200 個 5MB 檔
→ 一天 288,000 個檔案
```

### 正確做法 ✅

*   寫入目標檔案大小：
    *   **512MB – 1GB**
*   控制 Writer：
    *   Spark：`write.target-file-size-bytes`
*   **定期執行：**
    ```sql
    CALL system.rewrite_data_files(...)
    ```

📌 **經驗法則**：

> 「Iceberg 能承受 small files，但你不能忽視它們」

***

## 2️⃣ **Over‑Partitioning（老 Hive 思維）**

### 壞現象

```sql
PARTITIONED BY (year, month, day, hour, user_id)
```

### 為什麼是陷阱？

*   Iceberg **不是 Hive**
*   Partition ≠ 目錄剪裁
*   每個 partition 都增加 metadata cost

### Iceberg 的真強項

*   Hidden partitioning
*   Manifest‑based pruning
*   File‑level stats

### Recommended ✅

```sql
PARTITIONED BY (days(event_time))
```

或甚至：

```sql
NO PARTITION
```

📌 **反直覺真相**

> Iceberg 中「太多 partition」比「太少 partition」更糟

***

## 3️⃣ **High‑Frequency Commit（每筆就 commit）**

### 壞現象

*   每秒 commit 一次
*   Snapshot 成長超快
*   Catalog 熱點、寫入衝突

### 為什麼？

Iceberg commit 是：

    CAS compare-and-swap
    → metadata.json
    → catalog update

不是免費操作。

### 壞例子

*   REST API 每筆資料都觸發寫入
*   Streaming engine batch size = 1

### 正確做法 ✅

*   Batch writes
*   Streaming：
    *   micro‑batch > 100k rows
    *   或基於時間（1–5 min）
*   控制 snapshot 數量

***

## 4️⃣ **Streaming = 無限 Append（但沒觀念）**

### 壞現象

*   Kafka → Iceberg 直接寫
*   無 compaction
*   一週後 query 幾乎不能跑

### Iceberg Streaming 真相

Iceberg ≠ Streaming Store  
它是 **streaming‑friendly table format**

### 正確心法 ✅

Streaming pipeline = **三段式**

1.  ingest (append)
2.  **rewrite\_data\_files**
3.  expire\_snapshots

```text
Streaming writes
↓
Small files
↓
Scheduled compaction
```

> ❌ 沒 compaction = 設計錯誤

***

## 5️⃣ **多 Writer 沒規範（Concurrency Hell）**

### 壞現象

*   Spark job A 寫
*   Flink job B 寫
*   成功率低、commit fail

### 為什麼？

*   Iceberg 用 optimistic concurrency
*   同 table 同時寫 → 高機率衝突

### Anti‑pattern

    ETL job
    + Ad‑hoc notebook
    + Streaming job
    → 同一張 table

### 正確做法 ✅

*   **一張 table 一種 writer 類型**
*   或：
    *   append‑only table
    *   merge table 分離
*   使用 REST / managed catalog

***

## 6️⃣ **濫用 UPDATE / DELETE / MERGE**

### 壞現象

```sql
DELETE FROM table WHERE date < '2023-01-01';
```

### 為什麼危險？

Iceberg 的 row‑level ops：

*   equality delete file
*   position delete file

讀取時要：

    data file + delete files

會造成：

*   read amplification
*   metadata 複雜度上升

### 正確策略 ✅

*   能 **rewrite data files** 就 rewrite
*   大範圍修改：
    *   CTAS
    *   INSERT OVERWRITE（Iceberg 狀態）

📌 原則：

> Delete 是手術刀，不是推土機

***

## 7️⃣ **Schema Evolution 當成免費**

### 壞現象

*   任意 rename column
*   type 頻繁變更
*   同名不同意義

### Iceberg 是 schema‑on‑read，不代表：

*   consumer 免改
*   downstream 不炸

### 正確做法 ✅

*   schema change = API contract
*   rename ≠ drop + add（語意不同）
*   versioned rollout

***

## 8️⃣ **Metadata 無限成長（忘了做 housekeeping）**

### 常被忽略的大坑

*   snapshots
*   manifests
*   manifest lists

### 後果

*   plan time 超長
*   Driver OOM
*   Catalog latency

### 必做 ✅

```sql
CALL system.expire_snapshots(...)
CALL system.rewrite_manifests(...)
CALL system.remove_orphan_files(...)
```

📌 **Iceberg ≠ zero‑ops**

***

## 9️⃣ **選錯 Catalog，然後怪 Iceberg**

### 經典誤會

> 「Iceberg 寫入常 fail，不穩定」

實際上：

*   用 Hadoop catalog
*   S3 eventual consistency
*   多 writer

### 正確做法 ✅

*   POC：Hadoop catalog ✅
*   Production：
    *   REST
    *   Glue
    *   Managed catalog

Catalog 選錯 = 架構就錯一半

***

## 10️⃣ **把 Iceberg 當 OLTP**

### 壞期待

*   sub‑second write latency
*   high QPS random writes

### 現實

Iceberg 是：
✅ analytical table format  
❌ transactional DB

### 正確定位

*   ingestion layer ✅
*   analytical store ✅
*   serving store ❌
